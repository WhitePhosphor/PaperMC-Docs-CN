Velocity 与许多 Minecraft 服务器实现兼容。
如果服务器表现得像原版，Velocity 就会工作，并且我们尽可能为修改版服务器提供特殊支持。

## 兼容的游戏版本

截至本文撰写时，Velocity 兼容 Minecraft 1.7.2 至 {LATEST_MC_RELEASE}

## 原版设置

Velocity 最好与 Mojang 的原版服务器衍生的实现配合使用，这些实现不会向游戏本身添加内容。

### 原版服务器

Mojang 的原版软件处于一个复杂的境地。
它很有用，因为我们经常使用 Mojang 服务器进行协议更新测试，但在生产环境中，缺乏玩家信息转发支持可能会引发一些细微的客户端问题。

如果你打算运行一个原版服务器，**Velocity 团队强烈推荐你使用 Fabric 并搭配 FabricProxy-Lite 模组**。Fabric 和 FabricProxy-Lite 本身不会改变原版的游戏体验，你的服务器仍将与原版客户端兼容。
如果你无法（或不愿意）运行 Fabric，[VanillaCord](https://github.com/ME1312/VanillaCord) 可以让你使用传统的 BungeeCord 转发。

### Spigot

Spigot 并未经过与 Velocity 的充分测试。
然而，它基于原版，并且作为 Paper 的基础，它相对得到了较好的支持。

Spigot 不支持 Velocity 的现代转发功能，但支持传统的 BungeeCord 转发。

### Paper

Velocity 推荐使用 Paper 来运行一个公共服务器。Velocity 支持从 1.7.10 到最新版本的所有 Paper 版本。

如果你运行的是 Paper 1.13.2 或更高版本，你可以使用 Velocity 的现代转发功能。
如果你运行的是 Paper 1.12.2 或更低版本，你必须使用传统的 BungeeCord 的转发。

### SpongeVanilla

SpongeVanilla 与传统的 BungeeCord 风格转发和现代转发都兼容。
我们的 Sponge 支持主要集中在与 Forge 的兼容性上，更多信息请参见下文。

## 修改版设置

### Fabric

Velocity 与 Fabric 开箱即用，但你应该通过使用类似 [FabricProxy-Lite](https://modrinth.com/mod/fabricproxy-lite)（支持 Velocity 现代转发）的模组来添加对玩家信息转发的支持。

此外，如果你打算在原版的基础上运行添加新内容的模组，你应该安装 [CrossStitch](https://modrinth.com/mod/crossstitch)，
它改进了对某些被模组扩展的 Minecraft 功能的支持，例如自定义参数类型。
这个模组由 Velocity 项目官方维护。

### Minecraft Forge (1.13 and above)

截至 3.3.0 版本，Velocity 支持 1.20.2 以上版本的 Forge 服务器。
然而，对于 1.13 到 1.20.1 之间的中间版本，目前没有计划提供支持，不过你可以使用 [Ambassador](https://modrinth.com/plugin/ambassador) 插件来支持这些版本。

要为 Forge 添加现代转发支持，你必须安装与服务器版本相对应的 [ProxyCompatibleForge](https://modrinth.com/mod/proxy-compatible-forge) 或 [SpongeForge](https://spongepowered.org/downloads/spongeforge) 模组。
*请注意，SpongeForge 目前处于对 1.20.4 版本的 Forge 支持的实验阶段*。

### Minecraft Forge (1.7.2-1.12.2)

Minecraft Forge 适用于 Minecraft 1.7.2-1.12.2，与 Velocity 完全兼容，因为我们特别安排了同步客户端状态与每个服务器。
然而，我们**强烈**推荐使用 SpongeForge 或 BungeeForge，因为它允许你使用旧版 BungeeCord 玩家信息转发，并且总体上改善了代理支持。

Velocity 不支持 Forge-Bukkit 混合服务端——它们导致了许多问题，而且 Bukkit API 的设计排除了任何合理的模组支持的可能性。

## 反代理 (BungeeCord, Velocity, ...)

这些设置是**完全不被支持的**。你最好避免使用它们，因为它们可能会引发许多问题。
大多数反代理的设置，要么从一开始就毫无逻辑，要么可以通过更好、更可扩展且性能更高的解决方案更优雅地处理。

## 其他实现
